# Internationalization

Consult this file when a task touches localized routing, translated UI, locale helpers, or legal/error content.

## Routing And Locale Structure

- The site uses `astro-react-i18next` with French as the default locale and English as the secondary locale.
- French stays at `/`. English is prefixed under `/en`. Keep `prefixDefaultLocale: false` behavior intact.
- Keep localized routing static. Do not introduce runtime-only locale handling for page content.
- Localized page routes live under `src/pages/[...locale]/`. New localized pages should use `buildLocaleStaticPaths()` for static generation.
- Localized error pages live under `src/pages/[...locale]/errors/`. The root `src/pages/404.astro` is a French fallback page and should stay aligned with the French error experience.

## Locale Helpers And Runtime Setup

- Shared locale helpers live in `src/i18n/config.ts`. Reuse `normalizeLocale`, `buildLocaleStaticPaths`, `getLocalizedHref`, and related helpers instead of rebuilding locale logic ad hoc.
- `astro.config.mjs` imports `src/i18n/bootstrap.ts` before integrations are registered. Keep that bootstrap in place unless the i18n integration strategy is explicitly changed.
- Shared translation resources live in `public/locales/{fr,en}/` and are split by namespace: `common.json`, `seo.json`, and `errors.json`.
- Long-form legal content is localized through content collections and MDX in `src/content/legal/{fr,en}/`.

## React And i18n

- Prefer server-rendered React components for translated UI when possible.
- If a React component is rendered from an Astro page or layout, prefer passing `locale` and using `i18n.getFixedT(locale, namespace)` inside the component.
- Avoid hydrating large translated React subtrees just to switch language. Hydrate only the truly client-side behavior.
- If a component must remain client-only and locale-aware, keep the translated surface area narrow and use `src/components/hooks/useActiveLocale.ts` only for components that genuinely need runtime locale sync.
- For mixed static/client behavior, isolate the client-side behavior into a small island or DOM/CSS behavior instead of hydrating the entire translated section.

## Verification

- When changing localized routing or locale helpers, validate with `yarn build` because all locale routes are statically generated.
- When adding or editing translated UI copy, update both `public/locales/fr/` and `public/locales/en/` unless the task explicitly says otherwise.
- When adding localized pages, verify the generated routes for both locales, especially `/`, `/en`, `/terms`, `/en/terms`, and localized error pages when relevant.
- When editing SEO or error copy, check the matching namespace files in `public/locales/*/seo.json` and `public/locales/*/errors.json`.
- When editing MDX-backed legal content, keep both locale entries in `src/content/legal/fr/terms.mdx` and `src/content/legal/en/terms.mdx` aligned.
- If a change introduces client-side locale behavior in React, test client-side navigation between French and English to catch hydration mismatches early.
